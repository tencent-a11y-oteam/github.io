摘要

本规范定义了在 [HTML] 元素上使用 Accessible Rich Internet Applications (WAI-ARIA) 1.2 和 Digital Publishing WAI-ARIA Module 1.0 中相关属性的使用规则（创作时的一致性要求）。本规范的主要目标是定义作者（例如：web 开发人员）在使用一致性检查器时的要求。这些要求可以帮助作者使用 ARIA 来完善或扩展用 [HTML] 语言开发的 web 内容，包括自定义的界面或组件。

    1. 在 HTML 中使用 ARIA 的创作要求

    根据 WAI-ARIA 规范，作者可以使用 ARIA role 和 aria-* 属性来改变 HTML 元素对外暴露的含义（语义），除非所用特性与 strong native semantics 冲突或与 HTML 元素的隐式 ARIA 语义相同。HTML 的隐式 ARIA 语义由 HTML Accessibility API Mappings 规范定义。

    本规范定义的在 HTML 中使用 ARIA 功能的任何限制均是旨在防止辅助产品暴露与实际的页面 UI 不相符的信息。

    作者使用 ARIA role 和 aria-* 属性时，不得与4. 在 HTML 中使用 ARIA 属性的文档一致性要求 和4.2 使用 ARIA 属性代替等效 HTML 属性的要求冲突。不推荐将 ARIA role 和 aria-* 属性设置为上述两个规范中定义的隐式 ARIA 语义值，这样做不仅是多余的，而且还可能会导致意想不到的后果。

    2. 从 HTML 拓展与衍生的 ARIA 语义

    本节不具有规范性。

    通过使用 ARIA，作者可以赋予一个元素超出原生 HTML 功能的语义。这既能够提供作者创建 widget 的机会，又可以将某些仅使用 HTML 无法支持的无障碍状态和属性，暴露在原生 HTML 上。

    例如，一个 button 元素不能通过原生 HTML 特性来表示“按下”状态。ARIA 允许作者通过指定 aria-pressed 属性来扩展元素的语义 ，从而让听觉 UI 与控件的视觉表现相匹配。

    在以下示例中，一个 button 元素允许用户在 Web 应用中切换某项设置的状态。aria-pressed 属性用于扩展这个 button 元素。当元素处于“按下”状态时，此状态能够暴露给使用辅助阅读技术的用户。

    示例 1: 通过 ARIA 来表达按钮的按下状态 
    <button aria-pressed=true>...</button>

    在某些情况下，允许在具有特定 role 的元素上指定 aria-* 属性，但等效的 HTML 属性是无效的。

    例如，HTML 没有直接禁用超链接（a href）元素的概念。使用 <a href="..." disabled> ... </a> 是无效的，也不会传达给辅助技术。

    ARIA 在这方面与 HTML 不同，它允许在一个明确声明了 role=link 的元素上指定一个 aria-disabled 属性。如果作者在 HTML 超链接上指定属性 aria-disabled=true，用户代理在功能上不会对此超链接有任何不同的处理（仍然可点击/可操作），但它会被辅助技术认为处于禁用状态。

    类似地，HTML 中 option 元素是 select 元素的子元素，option 元素只能被设置为 selected，而具有明确 option 角色的元素不仅可以设置 aria-selected，还可以通过设置 aria-checked 属性来支持超出原生 select 元素能力的组件/结构。

    然而，当 ARIA 和 HTML 在功能上是相同的，但在适用范围上有差异时，可能会造成支持上的不一致，甚至是用户体验上的不一致。当 ARIA 允许 HTML 不支持的功能时，通常会为了作者和最终用户的最大利益，将其实现为一个完全自定义的 ARIA 组件。

    在下面的示例中，需要传达超链接的禁用状态。HTML 不允许在超链接上使用 disabled 属性，指定属性 aria-disabled=true 可以将超链接的禁用状态传达给辅助技术，但实际上并不能禁用该元素。 最有效的方法是去掉 a 元素中的 href 属性，再使用 ARIA 来传达该元素的预期作用和状态，这样既能传递禁用状态信息又能实际禁用超链接。

    示例2: 通过 ARIA 传达链接的禁用状态 
    <a role=link aria-disabled=true>...</a>

    3. 避免不正确使用 ARIA 的创作指南

    本节不具有规范性。

    3.1 避免在交互元素上声明非交互角色

    ARIA 可以用来修改或纠正一个元素的角色，以达到将该角色暴露给用户的效果。然而，使用 ARIA 覆盖交互元素的无障碍语义绝非最佳做法，例如，将非交互元素的角色值，如 heading，声明到交互元素 button 上。

    下面的示例在 button 元素上使用 role=heading。这是不被允许的，因为 button 元素的默认特性与 heading 角色的预期功能冲突。

    示例3：错误使用 role 
    <button role="heading">search</button>

    作者需要做额外的工作来消除 button 的默认特性和视觉呈现，甚至在用户与页面交互的其他场景，如开启 Windows 高对比度主题、开启浏览器阅读者模式下，作者无法根除元素自带的隐式特性。

    3.2 避免角色冗余

    下面的示例展示了一个显式指定 role=button 的 button 元素。显式指定此角色是多余的，因为 button 元素本身就隐式地对外暴露了 button 角色。实际应用中，角色冗余可能并不会造成无法预料的副作用，但是，这样不仅会使得代码冗长，而且还会误导其它的作者。 3.3 注意副作用小节可查阅添加冗余角色可能造成相关问题的示例。

    示例4：在 button 上添加冗余角色 
    <!-- 避免这种用法! -->
<button role="button">...</button>

    类似地，下面的示例在 fieldset 元素上添加了 role=group，在 main 元素上添加了 role=Main。这同样是冗余的，因为 fieldset 元素隐式地对外暴露了 group 角色，main 元素暴露了 main 角色。另外，在实际应用中，角色属性的值在声明时使用的是 ASCII 小写不会对使用者造成无法预料的副作用。有关详细信息，请查阅4.4 对 ARIA 角色、状态和属性值的书写要求。

    示例5：冗余角色 fieldset 和 main 
    <!-- 避免这种用法! -->
<fieldset role="group">...</fieldset>
<!-- 以及这种! -->
<main role="Main">...</main>

    下面的示例是在 ul 元素上添加了 role=list。 通常来说，这也是冗余的，因为 ul 元素本身就隐式地对外暴露了 role=list。但是如果移除了列表标记，一些用户代理厂商会忽略列表隐式暴露的 ARIA 语义。如有必要，作者可以添加 role=list 来显式暴露角色，但通常来说不推荐这种做法。

    示例 6：在 ul 上添加冗余角色 
    <!-- 通常应避免这种用法! -->
<ul role="list">...</ul>

    3.3 警惕副作用

    下面的示例是在 summary 元素上添加了 role=button，这同样是冗余的并且还可能导致跨平台问题。如阻止元素暴露正确的状态并强制暴露 button 角色，然而该元素原本在某平台或浏览器下有特定的角色。

    示例 7：意料之外的后果 
    <details>
<!-- 避免这种用法! -->
<summary role="button">more information</summary>
...
</details>

    3.4 遵守 ARIA 的规则

    Accessible Rich Internet Applications (WAI-ARIA) 1.2 定义了一系列不常用的角色。其中被归类为抽象角色的禁止使用。下面的示例展示了无效使用抽象角色 select 的情形，此情形下应该使用 combobox 角色。

    示例 8：禁止作者使用抽象角色 
    <!-- 禁止这种用法! -->
<div role="select" ...>...</div>

    ARIA 还定义了一个 generic 角色，旨在为那些没有明确 ARIA 语义的 HTML 元素提供对等的特性能力。例如，HTML 的 div 和 span 元素等。ARIA 并不推荐普通作者直接使用 generic 角色，因为它的目标用户是用户代理厂商的开发者。

    在下面的示例中，推荐作者用 div 代替 article 元素，而不是在 article 元素上添加 generic 角色。如果无法更改 HTML 元素，将其角色指定为 presentation 或 none 以去掉 article 的隐式角色。

    示例 9：不要将元素角色指定为 generic 
    <!-- 避免这种用法！ -->
<article role="generic" ...>...</article>

    另外，作者同时使用原生的 HTML 特性和与之等效 ARIA 特性时，原生的 HTML 特性优先级更高，参阅Conflicts with Host Language Semantics。

    在下面的示例中，作者使用了 HTML 元素 input type=checkbox 并声明了 aria-checked=true。但是，用户代理会忽略 aria-checked 属性而暴露表单控件的原生属性状态。

    示例 10: 隐式 checked 状态优先级高于显式 ARIA 属性 
    <!-- 避免这种用法！ -->
<input type="checkbox" checked aria-checked="false">

    3.5 遵守 HTML 的规则

    尽管 ARIA 可以改变 HTML 暴露给辅助技术使用者的特性，但这种改变并不会影响 HTML 底层解析和允许的内容模型。例如 div 允许作者在元素上声明任意角色值，但无法以违背 HTML 规则的方式使用。

    在下面的示例中，作者在 div 元素上声明了 link 角色。HTML 规定，超链接 link 元素（暴露 role=link）可以作为 p 元素的子代元素，但 div 却不能作为 p 元素的子代元素。

    示例 11: 以无效的 HTML 嵌套方式修改 ARIA 语义 
    <!-- 避免这种用法！ -->
<p>
  ... <div role=link tabindex=0>...</div> ... 
</p>

    HTML 解析器会将上述代码以如下方式输出：

    示例 12: 非预期代码渲染、有效替代方案 
    <!-- 前例代码的渲染结果如下 -->
<p>...</p>
<div role=link tabindex=0>...</div> 
... 
<p></p>

<!-- 在 p 元素中可以使用 span 元素 -->
<p>
  ... <span role=link tabindex=0>...</span> ...
</p>

    虽然本规范指出了每个 HTML 元素上可以声明的 ARIA 属性，但本例说明了即使可以声明特定角色，但其使用的上下文仍可以导致潜在的渲染和无障碍问题。

    4. 在 HTML 中使用 ARIA 属性的文档一致性要求

    下表展示了在 HTML 文档中每个元素使用 ARIA 代码的文档一致性要求。另外，表格标出了 HTML 元素的隐式 ARIA 语义。元素的隐式 ARIA 语义在 HTML AAM 规范中定义。

    下表每行第二个单元格所给出的 ARIA 语义是同行第一个单元格中语言特性（元素）的隐式 ARIA 语义（角色、状态和属性）。第三个单元格定义了元素可以使用的 ARIA 角色值和 aria-* 属性。如果第三个单元格包含术语 Any role，则意味着 role 值可以是任何角色值。但是不建议作者在元素上声明元素的隐式角色值、generic 角色和 deprecated by ARIA 的角色。如果第三个单元格包含术语 No role，则意味着作者不得覆盖 HTML 元素的隐式 ARIA 语义或元素的原生语义。

    WAI-ARIA 指出了具有 prohibited states and properties 要求的角色。在元素上显式地指定这些角色后，某些 WAI-ARIA 属性将禁止使用，隐式地暴露了这些角色值的元素也是如此。

    作者不得在具有 Naming prohibited 标识的元素上添加 aria-label 或 aria-labelledby 属性，除非该元素允许作者显式地用 WAI-ARIA 角色覆盖其隐式角色语义。更多信息请查阅 4.1 使用 ARIA 属性命名元素的要求。

    备注

    虽然并不推荐在元素上添加与隐式 ARIA 语义相同的 ARIA role 或 aria-* 属性，但在某些情况下显式地添加这些属性可能会有用，例如：用户代理不为特定元素暴露隐式 ARIA 语义。

    ARIA 规范中对已废弃特性的要求部分指出，如果某一项 ARIA 特性被标记为已废弃，则建议作者不要将其所述功能用于新内容。

    • directory
    备注

    directory 角色在 WAI-ARIA 1.2 中被标记为已废弃。实际使用中，对元素指定 directory 角色，相比隐式或显式指定 list 角色，本质上并无差异。建议作者选择使用 HTML 原生的列表元素，或使用 ARIA list 角色来代替。

    4.4 对 ARIA 角色、状态和属性值的书写要求

    所有的 role 值、被定义为 tokens 的属性值、状态值（aria-*）均应该为 ASCII 小写。

    备注

    尽管现代浏览器在对待 role 或 aria-* 属性值时 ASCII 大小不敏感，但并非所有的辅助技术都能够正确解析。

    为了减少互操作性问题，强烈建议作者在使用 aria-* 和 role 属性值时均使用 ASCII 小写。同时，为了确保内容能够正确地展示给用户，建议作者使用不同的浏览器和辅助技术相组合的方式进行严格的测试。

    6. 一致性

    除明确标记为非规范的部分外，规范中所有的开发指南、图表、示例和备注也不具有规范性。其余的内容都是具有规范性的。

    文档中出现的关键词 MAY，MUST，MUST NOT，NOT RECOMMENDED，SHOULD，和 SHOULD NOT，当且仅当它们以全大写字母出现时，才按照 BCP 14[RFC2119] [RFC8174] 所述的进行解释。

    6.1 一致性检查要求

    支持在 HTML 文档中检查 ARIA 的一致性检查器必须按照规范中的定义对 HTML 元素上使用 ARIA role 和 aria-* 属性的一致性要求进行检查。

    若元素的 role值或 aria-* 属性值是作者自定义的值而非4. 在 HTML 中使用 ARIA 属性的文档一致性要求中作者可以在 HTML 元素上使用的值，则符合规范的文档不得包含此元素。如果作者在元素上显式指定了与其隐式 ARIA 语义等效的角色，一致性检查器应该将实例标记为失败，因为不建议这样做。

    文档不符合规范时，一致性检查器可以自定义对应的术语和严重程度。

    7. 隐私和安全事项

    本节不具有规范性。

    本规范未定义 [wai-aria-1.2]、[dpub-aria-1.0] 或 [HTML] 的相关特性。但本规范为在 HTML 文档中检查 ARIA 的一致性检查器提供了规则和指南，同时也为作者提供了指南。

    因此，规范没有已知的隐私或安全风险，因为规范未定义可引入潜在问题的新特性。
